# 紧急修复 (/hotfix)

生产环境紧急问题的快速修复流程。

## 使用方式

```
/hotfix [紧急问题描述]
```

## 执行流程

### 1. 直接修复
- 跳过规划阶段
- 快速定位问题并修复
- 仍然需要理解根因（不能随意猜测）

### 2. 补测试
- 修复后补写回归测试
- 确认修复有效且有测试保护

### 3. 验证
- 执行 `/verify` 确认修复
- 重点检查：构建、类型、现有测试

### 4. 立即提交
- Commit message 格式：`fix: [紧急修复描述]`
- 建议在 hotfix 分支上操作

## 与 /bugfix 的区别

| | /bugfix | /hotfix |
|---|---------|---------|
| 规划 | Quick 模式 | 无规划 |
| 调试 | 完整系统化调试 | 快速定位 |
| 测试 | 先写测试再修复 | 先修复后补测试 |
| 节奏 | 稳健 | 紧急 |

## 重要提醒

- Hotfix 只用于真正的紧急情况
- 不紧急的 bug 应使用 `/bugfix`
- Hotfix 完成后安排时间回顾，避免同类问题再次发生
